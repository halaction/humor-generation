\section{Current results}

The implemented pipeline already supports corpus collection, embedding generation, keyword extraction, prompt construction, reference retrieval, candidate generation, and pairwise automatic evaluation. The remaining missing stage is the actual reinforcement-learning update loop. For that reason, this section is narrow: it reports the completed data artifacts and states the concrete experimental setting for the first MRVF training runs.

The corpus-level result is a merged joke collection of approximately \(664\text{k}\) items. In the final training setup, this full corpus serves as the retrieval pool: all jokes are embedded and indexed, and references for weak prompts are retrieved from this full embedding space. Keyword extraction, however, only needs to be applied to a sufficiently large subset to hit the target sample size. In our case, \(3-5\text{k}\) keyword groups can produce \(10-20\text{k}\) references for training, validation and testing.

This is all for the preliminary stage. The important result is that the data and the experimental setting are now finalized: the corpus exists, the reference collection is implemented, the methodology is finalized, and the remaining work is confined to the MRVF training loop.

\section{Planned training procedure}

The repository already implements the full data pipeline in Python using Hugging Face Datasets, parquet artifacts, FAISS-based nearest-neighbor search, and OpenAI-compatible API calls for embedding generation, candidate generation, and automatic evaluation. For the actual training runs, however, the software stack will change slightly.

The planned implementation will use a patched version of \texttt{trl} for optimization and \texttt{vLLM} for generation. \texttt{trl} provides the trainer-side logic for policy optimization, while \texttt{vLLM} will be used both to generate the baseline outputs and to serve rollouts during reinforcement learning. Since MRVF depends on sampling multiple reasoning traces per prompt, rollout throughput matters for efficiency.

The target models are from the Qwen3 family in the \(1.7\text{B}\) and \(4\text{B}\) size range. For each size, the planned baselines are:
\begin{enumerate}[leftmargin=1.8em]
    \item \textbf{Qwen3 Instruct}, baseline in non-thinking mode;
    \item \textbf{Qwen3 Thinking}, baseline in thinking mode;
    \item \textbf{MRVF-trained Qwen3 Thinking}, proposed model;
\end{enumerate}

This setup keeps the comparison interpretable. It isolates three distinct questions: whether ordinary instruction tuning is enough for humor generation, whether native thinking mode already improves humor generation without additional training, and whether MRVF can further improve the thinking model.

All training runs are planned for 1-2 \texttt{NVIDIA H200} GPUs on the HSE HPC cluster. This is sufficient for the intended model sizes and allows for parallelization accross instances.

The comparison will use the evaluation pipeline that is already implemented in the repository. Outputs are paired for the same prompt, the left/right order is randomized, the judge is forced to choose exactly one winner, and model scores are aggregated with a Bradley-Terry leaderboard. This is sufficient for the experiment because the goal is to detect whether MRVF produces a measurable shift relative to the instruct and thinking baselines. A larger multidimensional evaluation with rubric-based scoring and human annotation is planned after training is completed. The immediate objective is narrower: to test whether the proposed procedure can improve pairwise humor preference under a controlled prompt set.

\section{Planned ablations}

The ablation plan is directly connected to the theory. The most informative ablations are the following:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{Number of observed references per prompt \(|Y^{*}|\).} Compare smaller and larger reference sets, for example by truncating retrieval to \(1\), \(3\), or \(5\) items.
    \item \textbf{KL coefficient \(\beta\).} Test how strongly the model must be anchored to the initialization checkpoint in order to fix incompleteness.
    \item \textbf{Group size \(G\).} Vary the number of sampled reasoning traces per prompt in order to assess the variance / throughput trade-off of the MRVF estimator.
    \item \textbf{Model size.} Compare \(1.7\text{B}\) versus \(4\text{B}\), and check whether MRVF is robust when applied to smaller models.
\end{itemize}
